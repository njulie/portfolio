Object Storage Product Design Owner

Objects is one product in Nutanix's suite of storage service solutions. The first release allowed IT administrators to create object stores (databases) to begin storing photos, videos, documents, and other types of data files. I was the lead designer on this product, seeing the product through from beginning to early access.

The Problem

In a world where data is growing at an exponential rate, IT admins have to keep up. Traditional storage systems aren't equipped to handle petabytes of data, but object storage is.

Objects was designed to give IT admins a scalable solution that their teams and developers can use to store data, retrieve files quickly, protect against accidental file deletion and archive data to satisfy legal compliance (useful for finance, healthcare, and legal organizations).

The Users

Service Admin

Service admins (e.g. Northrop Grumman) help IT teams set up infrastructure services and may help other teams set up object storage.

IT admin

Many of our IT admin customers wanted object storage because it's cost-effective and will satisfy legal compliance use cases to maintain an archive.

App Developer

App developers will create "buckets" (like folders in Google Drive) to store their own objects without infringing upon other developer files.

Focus on the User, Less on Features

In the beginning of the design process, I received a list of engineering requirements of what features the team needed/wanted to build. I worked constantly with the Product Manager and tech leads to clarify what each technical requirement meant for the admin, reframing each requirement into a task that was done from the admin's perspective.

Sample Technical Requirement

On versioned buckets, delete non-current versions after x days

Sample User Story

To free up capacity, an admin/developer needs to set an automatic versioning policy that will delete old versions of a file after a set period of time.

I sifted through the list of prioritized engineering requirements and established a user story for each to ensure that what we were building aligned with a genuine user need.

Sample User Workflow: Create an Object Store

Design Iteration 1

The admin has to decide how much storage they need to allocate and what kind of performance they want to achieve. Storage for archived data typically have a lower performance, while storage that has data accessed often needs to have a higher level of performance.

View sample of high-fidelity mockups

Design Iteration 2

The first design was based upon the assumption that admins would like to see how changing one input (e.g. capacity) would impact another input (e.g. performance). However, we learned that as the capacity increases, the performance doesn't necessarily increase.

For example, admins may want to allocate a lot of storage and have performance be low if they intend to store archived data that doesn't need to be frequently accessed. As a result, I separated capacity and performance into two separate sections.

Sample User Workflow: Satisfy Legal Compliance

Design Iteration 1

I designed this under the assumption that admins/developers would want to create a folder (bucket is the technical term) initially dedicated to storing files for archival purposes. This bucket would contain photos, videos, documents and other media files that are unalterable. As such, in the workflow to create a bucket, I included the WORM setting (technical term for archival purposes) - only to be separated out after user testing with 5 customers.

Design Iteration 2

After testing the workflow with five customers, I learned that admins/developers don't really think about archival use cases when they create a bucket. They only get the legal requirement later in the process when they've already created the bucket and have started storing files in it. As such, it's not appropriate to incorporate the WORM section in the creation workflow, so I separated it into its own workflow.

Sometimes...explorations are dead ends

Sample Visuals

Early in the design process, I explored some ways to help admins get an overview of what types of objects/files are stored in their environment. We assumed that seeing a distribution of data by their size and where (what clusters) the data resided on would help admins better understand their storage environment and troubleshoot issues.

These explorations were scrapped after conducting a user test with five different admins. I discovered that while they understood what the visualizations were trying to convey, they expressed that they didn't know what to do with the information. In their day-to-day use of the product, they wouldn't find any value in seeing the visuals. While the engineers on the team were enthusiastic towards the visuals due to the "wow" factor, I decided it was better to keep the user interface filled with elements and actions admins would actually need on their day-to-day jobs rather than wow them once and leave them with irrelevant visuals on every day after.

Final Results

Feedback

After the early access release to around our initial customers, the team received feedback about how simple the object store experience was, compared to the typical complexity of storage infrastucture products.

Next Steps

Future improvements would include exploring more graphs that would be categorized by users so that admins could contact their developers when capacity was running low. Being able to see who was using how much storage would help with their internal reports.
